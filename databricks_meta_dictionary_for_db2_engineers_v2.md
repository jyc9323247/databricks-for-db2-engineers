# Databricks 데이터 엔지니어 메타 사전 v2.0 (DB2 엔지니어 대상)

> **문서 목적**: DB2 환경에서 근무하던 엔지니어들이 Databricks 레이크하우스(Lakehouse) 환경으로 기술 전이를 할 수 있도록, 기존 RDBMS(DB2) 개념과 매칭하여 핵심 용어를 정리한 메타 사전입니다.
>
> **⚠️ 표기 안내**: 비교가 100% 정확하지 않거나 해석에 여지가 있는 항목은 `[⚠ 근사 비교]`로 표시했습니다. 이 항목들은 개념적 이해를 돕기 위한 비유이며, 내부 동작이 완전히 동일하지는 않습니다.

## 1. 아키텍처 및 데이터 모델링 (Architecture & Modeling)

| # | 용어 | 정의 | DB2 비교 및 이해 |
| :---: | :--- | :--- | :--- |
| 1-1 | **메달리온 아키텍처 (Medallion Architecture)** | 데이터 품질에 따라 Bronze → Silver → Gold 3단계로 관리하는 계층 설계 패턴. | Staging(Bronze) → ODS/EDW(Silver) → Data Mart(Gold) 흐름과 유사. |
| 1-2 | **데이터 레이크하우스 (Data Lakehouse)** | 데이터 레이크의 유연성(비정형/반정형 수용)과 DW의 ACID 트랜잭션·성능을 결합한 통합 아키텍처. | DB2의 정형 데이터 처리 능력에 비정형 데이터 저장 능력이 합쳐진 형태. |
| 1-3 | **관리형 테이블 (Managed Table)** | Unity Catalog가 데이터 파일의 저장 위치와 생명주기를 모두 관리하는 테이블. DROP 시 데이터도 함께 삭제. | DB2의 일반 테이블(Regular Table)과 동일. DROP TABLE 시 데이터가 사라지는 것과 같음. |
| 1-4 | **외부 테이블 (External Table)** | 데이터 파일이 사용자가 지정한 외부 경로(S3, ADLS)에 존재하며, DROP 시 메타데이터만 삭제. | `[⚠ 근사 비교]` DB2의 Federation Nickname과 "외부 데이터를 참조"한다는 점이 유사하지만, External Table은 로컬 SQL로 직접 읽고 쓰는 반면 Nickname은 Wrapper를 통해 리모트 DB에 SQL을 전달하는 구조. 개념적으로는 DB2에서 외부 데이터 파일을 참조하도록 DDL만 정의한 테이블에 더 가까움. |
| 1-5 | **파티셔닝 (Partitioning)** | 특정 컬럼 값 기준으로 데이터를 물리적 디렉토리(Hive-style)로 분할 저장. | DB2의 Table Partitioning(Range Partitioning)과 목적이 동일. 다만 Databricks는 디렉토리 기반, DB2는 Data Partition 기반이라는 물리적 차이 존재. |
| 1-6 | **Z-오더링 (Z-Ordering)** | 지정한 컬럼들의 값 범위가 겹치지 않도록 데이터를 파일 단위로 재배치하는 다차원 클러스터링 기술. | DB2의 MDC(Multidimensional Clustering) 테이블과 가장 유사. 차이점: MDC는 INSERT 시 자동 배치, Z-Order는 OPTIMIZE 명령으로 사후 재배치. |
| 1-7 | **리퀴드 클러스터링 (Liquid Clustering)** | 정적 파티셔닝과 Z-Ordering을 대체하는 차세대 동적 데이터 배치 기술. 클러스터링 키를 사후에 변경 가능하며, OPTIMIZE 시 점진적으로 데이터를 재배치. | `[⚠ 근사 비교]` 데이터 분포에 따라 자동으로 셀 크기를 조절하는 '지능형 MDC'로 이해 가능. 다만 DB2 MDC와 달리 키 변경이 가능하고, 백그라운드 점진 처리라는 근본적 차이 존재. |
| 1-8 | **스키마 진화 (Schema Evolution)** | 새 데이터 적재 시 기존 테이블 스키마에 없는 컬럼을 자동으로 추가(ADD COLUMN)하거나 타입을 확장하는 기능. `mergeSchema` 옵션으로 제어. | DB2의 엄격한 Schema-on-Write에서는 불가능한 기능. DB2에서는 반드시 DDL(ALTER TABLE)을 먼저 수행해야 함. |
| 1-9 | **스키마 강제 (Schema Enforcement)** | 기존 테이블 스키마와 일치하지 않는 데이터 쓰기를 거부하는 Delta Lake 기본 동작. | DB2의 기본 동작(INSERT 시 컬럼 타입/개수 불일치 → 에러)과 동일한 원칙. |
| 1-10 | **구체화된 뷰 (Materialized Views)** | 쿼리 결과를 물리적으로 저장하고 소스 변경 시 점진적으로 갱신하는 뷰. DLT 파이프라인 내에서 또는 SQL Warehouse에서 사용. | DB2의 MQT(Materialized Query Table)와 거의 일치하는 개념. `REFRESH TABLE` ↔ `REFRESH DEFERRED`. |
| 1-11 | **테이블 클론 (Table Clones)** | **Shallow Clone**: 메타데이터·트랜잭션 로그만 복제(원본 데이터 파일 참조). **Deep Clone**: 물리 파일까지 전체 복제. | Shallow는 스토리지 스냅샷, Deep은 `CREATE TABLE ... AS SELECT`(CTAS)와 유사. |
| 1-12 | **OPTIMIZE** | 소규모 파일들을 적정 크기(~1GB)로 병합(Compaction)하여 읽기 성능을 개선하는 명령. | DB2의 `REORG TABLE`과 동일한 관리 작업. 단편화(Fragmentation) 해소 목적. |
| 1-13 | **VACUUM** | 더 이상 참조되지 않는 과거 버전의 데이터 파일을 물리적으로 삭제하여 스토리지를 회수하는 명령. 기본 보존 기간 7일. | DB2의 아카이브 로그 정리(`PRUNE HISTORY`) 또는 `RUNSTATS` 후 불필요 임시 파일 정리와 유사한 관리 작업. |
| 1-14 | **예측적 최적화 (Predictive Optimization)** | Unity Catalog가 테이블 사용 패턴을 분석하여 OPTIMIZE, VACUUM, ANALYZE를 자동 스케줄링하는 기능. | DB2의 Auto Maintenance(Automatic RUNSTATS, REORG)와 동일한 자동화 관리 체계. |
| 1-15 | **임시 뷰 (Temp View)** | `CREATE TEMP VIEW`로 생성하며 SparkSession(노트북 실행 단위) 동안만 유지되는 논리적 뷰. 물리적 데이터 저장 없음. | `[⚠ 근사 비교]` DB2의 DGTT(Declared Global Temporary Table)와 "세션 범위"라는 점은 유사하나, DGTT는 물리적 데이터를 담는 테이블이고, Temp View는 쿼리 정의만 보관하는 뷰. 정확히는 DB2의 `WITH ... AS (CTE)` 또는 세션 내 인라인 뷰에 더 가까움. |
| 1-16 | **레이크하우스 페더레이션 (Lakehouse Federation)** | PostgreSQL, MySQL, SQL Server 등 외부 DB 데이터를 복사 없이 직접 쿼리하는 기능. Unity Catalog 내 Foreign Catalog로 등록. | DB2의 핵심 기능인 Federation(Wrapper → Server → Nickname)과 동일한 목적. |
| 1-17 | **SCD (Slowly Changing Dimension)** | 시간에 따라 변하는 차원 데이터의 이력을 관리하는 모델링 패턴(Type 1/2/3 등). DLT의 `APPLY CHANGES INTO`로 SCD Type 1/2를 자동 구현 가능. | DB2의 Temporal Table(System-Time / Business-Time) 및 이력 테이블 설계와 동일한 목적. |
| 1-18 | **자동 증가 컬럼 (Identity / Auto Increment)** | `GENERATED ALWAYS AS IDENTITY` 또는 `BIGINT GENERATED BY DEFAULT AS IDENTITY`로 순차 고유 키 자동 생성. | DB2의 `GENERATED ALWAYS AS IDENTITY`와 문법·동작 모두 거의 일치. |
| 1-19 | **볼륨 (Volumes)** | Unity Catalog에서 관리하는 비테이블 파일 저장소. 원시 파일(CSV, JSON, 이미지 등)을 거버넌스 하에 보관. | DB2에는 직접 대응 개념 없음. OS 파일 시스템의 특정 디렉토리를 DB 권한 체계로 관리하는 것으로 이해. |
| 1-20 | **델타 셰어링 (Delta Sharing)** | Databricks 외부 조직에게 데이터를 안전하게 공유하는 개방형 프로토콜. 데이터 복사 없이 읽기 전용 접근 제공. | DB2에는 직접 대응 기능 없음. Federation + 보안 정책의 조직 간 확장 버전으로 이해. |

## 2. 데이터 저장소 및 포맷 (Storage & Formats)

| # | 용어 | 정의 | DB2 비교 및 이해 |
| :---: | :--- | :--- | :--- |
| 2-1 | **파켓 (Parquet)** | 분석 쿼리에 최적화된 컬럼 지향(Columnar) 바이너리 저장 포맷. 컬럼 단위 압축, Predicate Pushdown, 컬럼 프루닝 지원. | Row 단위 저장인 DB2 기본 테이블과 달리, 열 단위로 압축 저장하여 집계 쿼리 성능 극대화. DB2 BLU의 컬럼 스토어와 저장 철학이 동일. |
| 2-2 | **델타 레이크 (Delta Lake)** | Parquet 파일 위에 JSON 기반 트랜잭션 로그(`_delta_log/`)를 추가하여 ACID 트랜잭션, Time Travel, Schema Enforcement 등을 보장하는 오픈소스 스토리지 계층. | DB2의 Transaction Log(Active/Archive Log)가 테이블 데이터의 일관성을 보장하듯, Delta Log가 파일 레벨에서 동일한 역할 수행. |
| 2-3 | **델타 로그 (Delta Log / Transaction Log)** | `_delta_log/` 디렉토리에 JSON 파일로 기록되는 모든 테이블 변경 이력. 10 커밋마다 체크포인트(Parquet) 생성. | DB2의 Active Log + Archive Log와 목적이 동일(커밋 기록, 롤백 지원, 복구). |
| 2-4 | **타임 트래블 (Time Travel)** | Delta Log를 기반으로 테이블의 과거 버전 데이터를 조회하거나 복원하는 기능. `SELECT * FROM t VERSION AS OF 5` 또는 `TIMESTAMP AS OF '2024-01-01'`. | DB2에서는 Point-in-Time Recovery(PITR)로 DB 전체를 복원해야 하는 작업을 테이블 단위로 간편하게 수행. |
| 2-5 | **RESTORE** | Time Travel을 이용해 테이블을 특정 과거 버전으로 되돌리는 명령. `RESTORE TABLE t TO VERSION AS OF 3`. | DB2의 `ROLLFORWARD TO PIT` 작업을 테이블 단위로 단순화한 것. |
| 2-6 | **DESCRIBE HISTORY** | 테이블의 전체 변경 이력(버전, 타임스탬프, 수행 작업, 사용자 등)을 조회하는 명령. | DB2의 `db2 list history` 명령 + `SYSIBMADM.DB_HISTORY` 뷰와 유사한 감사 추적 기능. |
| 2-7 | **변경 데이터 피드 (Change Data Feed, CDF)** | Delta Table에서 행 단위 변경 내역(INSERT/UPDATE/DELETE 전후 값)을 별도 컬럼(`_change_type`)과 함께 읽을 수 있는 기능. `ALTER TABLE SET TBLPROPERTIES (delta.enableChangeDataFeed = true)`. | `[⚠ 근사 비교]` DB2의 CDC(IIDR 등)와 "변경분 추적"이라는 목적은 동일하나, CDF는 외부 에이전트 없이 Delta Lake 내장 기능으로 동작하는 차이. |
| 2-8 | **클라우드 오브젝트 스토리지** | S3, ADLS Gen2, GCS 등 무한 확장이 가능한 비공유형(Shared-Nothing) 저장소. | 용량을 사전 할당해야 하는 DB2 Table Space와 달리 무제한 확장 가능한 가상 디스크. 비용도 사용량 기반. |

## 3. 컴퓨팅 및 처리 엔진 (Compute & Processing Engine)

| # | 용어 | 정의 | DB2 비교 및 이해 |
| :---: | :--- | :--- | :--- |
| 3-1 | **아파치 스파크 (Apache Spark)** | 메모리 기반의 대규모 분산 처리 엔진. Python(PySpark), SQL, Scala, R 멀티 언어 지원. | DB2의 쿼리 실행 엔진이 여러 대의 서버로 분산된 형태. 단, SQL뿐 아니라 프로그래밍 API로도 데이터 처리 가능. |
| 3-2 | **Databricks Runtime (DBR)** | Apache Spark + Delta Lake + 라이브러리 + 최적화가 패키징된 실행 환경. 버전별로 Spark 버전이 매핑됨(예: DBR 15.x → Spark 3.5.x). | DB2 Fix Pack + Feature Pack이 묶여 있는 DB2 Server 빌드와 유사한 개념. |
| 3-3 | **클러스터 (Cluster)** | Driver(마스터 노드)와 Worker(실행 노드)로 구성된 연산 자원 집합. | DB2 DPF 환경의 Coordinator Node(Driver)와 Data Node(Worker) 구조와 유사. |
| 3-4 | **범용 클러스터 (All-Purpose Cluster)** | 노트북에서 대화형 개발·테스트용으로 사용하는 상시 구동 클러스터. 여러 사용자가 공유 가능. | 개발자들이 상시 접속하는 개발 DB2 인스턴스와 유사. |
| 3-5 | **작업 클러스터 (Job Cluster)** | Workflow의 특정 Task 실행 시에만 자동으로 생성되고 완료 후 즉시 종료되는 일회성 클러스터. | 배치 Job 실행 시간에만 기동하고 끝나면 꺼지는 서버. DB2에는 없는 클라우드 고유 개념. |
| 3-6 | **서버리스 컴퓨팅 (Serverless Compute)** | 클러스터 구성 없이 Databricks가 자동으로 컴퓨팅 자원을 프로비저닝하는 방식. 노트북, Workflow, DLT에서 사용 가능. | DB2에는 없는 개념. 서버 설치·관리 없이 SQL을 제출하면 자동으로 실행되는 것. |
| 3-7 | **카탈리스트 옵티마이저 (Catalyst Optimizer)** | Spark SQL의 논리적·물리적 쿼리 최적화 엔진. 규칙 기반 + 비용 기반 최적화 수행. | DB2의 SQL Optimizer와 동일한 역할. 다만 DB2는 더 성숙한 비용 기반 최적화(Cost-Based), Catalyst는 규칙 기반(Rule-Based)이 상대적으로 강함. |
| 3-8 | **포톤 엔진 (Photon Engine)** | C++로 구현된 벡터화(Vectorized) 네이티브 쿼리 실행 엔진. Spark JVM 엔진 대비 대폭적인 성능 향상. | DB2 BLU Acceleration(컬럼 기반 벡터 처리)의 클라우드 버전으로 이해 가능. |
| 3-9 | **적응형 쿼리 실행 (AQE)** | 쿼리 실행 도중 셔플 파티션 수, Join 전략(Sort-Merge → Broadcast), Skew 처리를 런타임 통계 기반으로 동적 재조정하는 기능. Spark 3.x 기본 활성. | `[⚠ 근사 비교]` 실행 중 스스로 Access Path를 재조정하는 '지능형 옵티마이저'. DB2 Optimizer는 컴파일 시점에 Plan을 확정하므로, 런타임 재조정이라는 점에서 근본적 차이. |
| 3-10 | **오토스케일링 (Autoscaling)** | 부하(Pending Task)에 따라 Worker 노드 개수를 자동으로 증감(min~max 범위)하는 기능. | Peak 용량에 맞춰 서버를 사두지 않고 필요할 때만 확장하는 클라우드 탄력성. DB2에는 없는 개념. |
| 3-11 | **SQL 웨어하우스 (SQL Warehouse)** | BI 도구 연동 및 ANSI SQL 쿼리에 최적화된 서버리스/전용 컴퓨팅 엔드포인트. JDBC/ODBC 접속 지원. | `[⚠ 근사 비교]` 분석 쿼리 전용으로 격리된 DB2 인스턴스 + WLM(Workload Manager) 역할. 다만 DB2의 WLM은 같은 인스턴스 내 리소스 분배, SQL Warehouse는 별도 컴퓨팅으로 완전 격리. |
| 3-12 | **스파크 UI** | Spark Job/Stage/Task 실행 계획, 셔플 크기, GC 시간 등 성능 병목 지점을 시각화하는 웹 대시보드. | DB2의 `db2expln`, `db2pd`, Visual Explain을 합친 모니터링 대시보드. |
| 3-13 | **쿼리 프로파일 (Query Profile)** | SQL Warehouse에서 실행된 쿼리의 상세 실행 계획, 각 연산자별 소요 시간·행 수를 시각화. | DB2의 `EXPLAIN` + Activity Monitor의 결합. |
| 3-14 | **접근 모드 (Access Mode)** | 클러스터의 보안 격리 수준을 결정하는 설정. **Single User**: 1명 전용, **Shared**: 다중 사용자 공유(Unity Catalog 강제). | `[⚠ 근사 비교]` DB2의 인스턴스 접근 권한 레벨(SYSADM, DBADM 등)과 유사하지만, Access Mode는 런타임 격리에 초점. |
| 3-15 | **스팟 인스턴스 (Spot Instances)** | 클라우드 사업자의 유휴 자원을 할인된 가격(최대 90%)으로 사용하되, 언제든 회수될 수 있는 Worker 노드. | DB2에는 없는 클라우드 고유 비용 최적화 개념. Worker가 회수되면 해당 Task만 다른 노드에서 재실행. |

## 4. 데이터 엔지니어링 및 파이프라인 (Pipelines & Ingestion)

| # | 용어 | 정의 | DB2 비교 및 이해 |
| :---: | :--- | :--- | :--- |
| 4-1 | **DLT (Delta Live Tables)** | 선언적(Declarative) 방식으로 데이터 파이프라인을 구축하는 프레임워크. 의존성 자동 해석, 오류 복구, 품질 검증 내장. | MQT + ETL 스케줄링 + Data Quality Check가 통합된 고급 프레임워크. DB2에는 직접 대응 없음. |
| 4-2 | **스트리밍 테이블 (Streaming Table)** | DLT에서 Append-Only 스트리밍 데이터를 수신하는 대상 테이블. `CREATE STREAMING TABLE`. | 실시간 INSERT만 발생하는 로그 적재 테이블과 유사. |
| 4-3 | **오토 로더 (Auto Loader)** | 클라우드 스토리지에 새로운 파일 도착을 자동 감지하여 증분 데이터만 효율적으로 로드하는 Structured Streaming 소스. 파일 알림(Notification) 또는 디렉토리 리스팅(Directory Listing) 모드 지원. | 파일 입고를 감지하여 자동으로 `LOAD`를 수행하는 지능형 로더. DB2의 `LOAD` + 쉘 스크립트 Polling 자동화 버전. |
| 4-4 | **COPY INTO** | 파일 기반 데이터를 테이블로 1회성 고속 적재하는 SQL 명령. 이미 적재된 파일은 건너뜀(멱등성). | DB2의 `LOAD` 유틸리티와 목적·사용법 유사. 대량 초기 적재에 적합. |
| 4-5 | **MERGE INTO** | 소스와 타겟의 조건 매칭에 따라 INSERT, UPDATE, DELETE를 한 문장으로 수행하는 DML. | DB2의 `MERGE INTO` 구문과 문법·동작 모두 거의 일치. |
| 4-6 | **APPLY CHANGES INTO (CDC)** | DLT 내에서 소스 시스템의 CDC 피드(INSERT/UPDATE/DELETE)를 자동으로 Delta Table에 병합. SCD Type 1/2 자동 지원. | DB2의 InfoSphere CDC(IIDR), Q Replication과 동일한 목적. DLT 프레임워크에 통합되어 별도 솔루션 불필요. |
| 4-7 | **품질 제약조건 (Expectations)** | DLT 파이프라인 내에서 데이터 품질 규칙을 선언하고, 위반 시 경고(warn) / 삭제(drop) / 실패(fail)를 선택하는 기능. | DB2의 `CHECK CONSTRAINT`와 유사하나, 불량 데이터를 격리(quarantine) 가능하다는 점에서 더 유연. |
| 4-8 | **구조적 스트리밍 (Structured Streaming)** | 실시간 스트림 데이터를 마이크로 배치 방식으로, 배치 SQL과 동일한 API로 처리하는 Spark 엔진. Kafka, Event Hub, Kinesis 소스 지원. | MQ 등 스트림 데이터를 복잡한 저수준 로직 없이 SQL/DataFrame으로 처리하는 통합 방식. |
| 4-9 | **DAG (Directed Acyclic Graph)** | 태스크 간의 선후행 의존성을 비순환 방향 그래프로 표현한 실행 흐름도. DLT, Workflows 모두 내부적으로 DAG 구성. | Control-M, TWS(Tivoli Workload Scheduler)의 Job Dependency Tree와 동일 개념. |
| 4-10 | **체크포인트 (Checkpointing)** | Structured Streaming에서 처리 진행 상태(offset, 상태 데이터)를 영구 저장소에 기록하여, 장애 시 정확히 해당 지점부터 재시작하는 메커니즘. | DB2의 LSN(Log Sequence Number)이나 배치 프로그램의 재시작(Restart/Checkpoint) 로직이 자동화된 것. |
| 4-11 | **트리거 (Trigger Modes)** | Streaming 쿼리의 실행 주기 설정: `availableNow`(현재 가용 데이터 모두 처리 후 종료), `processingTime`(주기적 마이크로배치), `continuous`(초저지연). | `availableNow` = 배치 Job, `processingTime` = 주기적 Polling, `continuous` = 실시간 데몬. |
| 4-12 | **워터마크 (Watermarking)** | Structured Streaming에서 지연 도착(Late-Arriving) 데이터를 허용하는 시간 임계값을 설정하는 메커니즘. | `[⚠ 근사 비교]` 비즈니스 마감(Closing) 로직을 시스템 레벨에서 처리하는 개념. DB2에는 직접 대응 없으며, 보통 Application 로직으로 구현하던 부분. |
| 4-13 | **파일 도착 트리거 (File Notification)** | Auto Loader의 파일 알림 모드에서 클라우드 이벤트(S3 SQS, ADLS Event Grid)를 수신하여 새 파일 도착 즉시 처리 시작. | 쉘 스크립트의 `ls` Polling 대신 이벤트 기반(Event-Driven)으로 배치를 시작하는 방식. |

## 5. 데이터 거버넌스 및 카탈로그 (Unity Catalog & Governance)

| :---: | :--- | :--- | :--- |
| 5-1 | **유니티 카탈로그 (Unity Catalog)** | 전사적 데이터 자산(테이블, 뷰, 볼륨, 모델, 함수)에 대한 통합 거버넌스·접근 제어·감사·리니지 솔루션. | DB2 시스템 카탈로그(SYSCAT) + 권한 관리(GRANT/REVOKE) 체계의 클라우드 확장판. |

| 5-4 | **데이터 리니지 (Lineage)** | Unity Catalog가 테이블·컬럼 간 데이터 흐름 경로를 자동으로 추적·시각화하는 기능. 별도 설정 불필요. | 외부 메타데이터 관리 솔루션(InfoSphere Information Governance Catalog 등)이 플랫폼에 내장된 형태. |

| 5-8 | **동적 뷰 (Dynamic Views)** | `current_user()`, `is_member()` 등 런타임 함수를 뷰 정의에 포함하여, 조회 사용자에 따라 다른 결과를 반환하는 뷰. | DB2의 RCAC + 뷰를 결합한 보안 뷰와 유사. 접속 사용자에 따라 데이터가 필터링/마스킹됨. |
| 5-9 | **태그 (Tags)** | 테이블, 컬럼 등 데이터 자산에 키-값 쌍의 메타데이터 태그를 부여하여 분류·검색·정책 적용을 용이하게 하는 기능. | DB2에는 직접 대응 없음. 별도 메타데이터 관리 시스템에서 수행하던 데이터 분류 작업을 카탈로그 내에서 처리. |

## 8. SQL 및 쿼리 고급 기능 (Advanced SQL Features)

| 8-5 | **파라미터화 쿼리 (Parameterized Queries)** | 이름 기반(`:param`) 또는 위치 기반(`?`) 매개변수로 쿼리를 동적 실행. SQL Injection 방지. | DB2의 Parameter Marker(`?`)를 사용한 Prepared Statement와 동일 목적. |
| 8-6 | **PIVOT / UNPIVOT** | 행을 열로 회전(PIVOT) 또는 열을 행으로 변환(UNPIVOT)하는 SQL 구문. | DB2 11.1+ 에서 지원되는 `PIVOT` / `UNPIVOT` 구문과 동일 (DB2 이전 버전에서는 CASE WHEN으로 수동 구현). |
| 8-7 | **제약 조건 (Constraints)** | `PRIMARY KEY`, `FOREIGN KEY`, `NOT NULL`, `CHECK` 선언 가능. 단, Delta Lake에서는 `NOT NULL`과 `CHECK`만 실제로 강제 적용(enforce)되며, PK/FK는 정보성(informational). | `[⚠ 근사 비교]` DB2에서는 모든 제약 조건이 실제로 강제 적용되지만, Databricks의 PK/FK는 쿼리 옵티마이저 힌트 역할만 수행하고 DML 시 검증하지 않음. 이 차이가 중요. |
| 8-8 | **UDF (User Defined Functions)** | Python 또는 SQL로 사용자 정의 함수를 작성하여 SQL 내에서 호출. Unity Catalog에 등록하여 전사적 공유 가능. | DB2의 UDF(SQL UDF, External UDF)와 동일 개념. 다만 DB2 External UDF는 C/Java, Databricks UDF는 Python/SQL. |

## 부록 B: 자주 혼동되는 개념 정리

| 혼동 포인트 | 설명 |
| :--- | :--- |
| **OPTIMIZE vs VACUUM** | `OPTIMIZE` = 소파일 병합(REORG), `VACUUM` = 구버전 파일 삭제(PRUNE). 두 작업은 독립적이며 모두 정기 수행 필요. |
| **Auto Loader vs COPY INTO** | Auto Loader = 지속적 증분 감지(Streaming 기반), COPY INTO = 1회성 배치 적재. 지속적 파이프라인은 Auto Loader, 초기 적재는 COPY INTO 권장. |
| **Managed vs External Table** | DROP 시 데이터 삭제 여부가 핵심 차이. Unity Catalog 환경에서는 가능한 한 Managed Table을 사용하여 거버넌스 일원화 권장. |
| **Partition vs Z-Order vs Liquid Clustering** | 카디널리티 낮은 컬럼(날짜 등) → Partition. 카디널리티 높은 컬럼 → Z-Order. 신규 테이블이면 Liquid Clustering 우선 검토(Partition + Z-Order 대체). |
| **PK/FK 제약 조건** | DB2와 달리 Databricks의 PK/FK는 **정보성(informational)**이며 DML 시 강제 적용되지 않음. 쿼리 옵티마이저 힌트와 문서화 목적으로만 활용. |
| **Job Cluster vs All-Purpose Cluster** | 비용 관점에서 운영 파이프라인은 반드시 Job Cluster 사용(실행 시에만 과금). All-Purpose는 개발·디버깅 전용. |
| **DLT vs Workflows** | DLT = 데이터 파이프라인 전용(선언적, 품질 검증 내장). Workflows = 범용 오케스트레이션(DLT 실행도 Task로 포함 가능). |
| **CDF(Change Data Feed) vs CDC(Apply Changes Into)** | CDF = Delta Table 자체의 변경분 추적 기능. CDC(APPLY CHANGES INTO) = 외부 소스의 변경분을 DLT로 수신·병합하는 기능. |
